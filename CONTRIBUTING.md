# Contributing to `ironclaw-mcp`

Thanks for your interest in contributing. `ironclaw-mcp` is a generic Go MCP
server that bridges AI coding assistants to an
[IronClaw](https://github.com/nearai/ironclaw) HTTP gateway. The project aims
to stay deployment-agnostic: every default tool must speak only the
documented IronClaw HTTP API and ship no opinionated assumptions about a
particular fleet, persona catalogue, or auxiliary CLI.

## Project ground rules

- **Generic only.** Tools that are specific to a particular IronClaw
  deployment (for example fleet ops, persona spawn, or workspace
  integrations) belong in a sibling MCP server, not in `ironclaw-mcp`.
- **No internal references.** Source, tests, and docs must not reference
  private home-directory paths, private repos, named personas, or sprint
  markers. The repository is a public artefact.
- **English: American spelling** in code and changelog. Either British or
  American is acceptable in commit messages and PR descriptions.
- **Conventional Commits.** Use `feat:`, `fix:`, `docs:`, `refactor:`,
  `test:`, `chore:`, `ci:` prefixes. Scope is optional but encouraged
  (e.g. `feat(tools): add ironclaw_export_session`).
- **No AI attribution.** Do not add "Co-authored-by: AI", "Generated by …",
  or similar trailers to commits.

## Local development

### Prerequisites

- Go 1.24+
- A running IronClaw gateway on `http://localhost:3000` (for the smoke flow)

### Quick loop

```bash
# Tidy + fmt + vet + lint + test (all gates)
make check

# Coverage report
make coverage

# Local smoke against a running IronClaw gateway
make smoke
```

### Test policy

- All new packages must ship table-driven unit tests.
- HTTP client/handler changes must use `httptest.NewServer` integration
  tests; do not add tests that depend on a live IronClaw instance.
- The CI gate is **70% total coverage** and rising. PRs that drop total
  coverage below the existing baseline will not be merged.
- `go test -race -count=1 ./...` must pass cleanly. Flakes block merge.

### Adding a tool

1. Implement the handler in `internal/tools/<domain>.go` with a matching
   `<domain>_test.go`.
2. Register the handler in `internal/server/server.go`. Generic IronClaw
   gateway tools register unconditionally; deployment-specific tools do not
   belong in this repository.
3. Update the **default tool surface** table in `README.md`. The tool count
   is asserted in `internal/server/server_test.go`; bump that assertion in
   the same PR.
4. Add a `CHANGELOG.md` entry under `## [Unreleased]`.

### Adding configuration

1. Document the env var in `internal/config/config.go` and add a parser
   helper if the value is not a plain string.
2. Add a unit test in `internal/config/config_test.go` covering the
   default + override + invalid cases.
3. Add the variable to the **Configuration** table in `README.md`.

## Pull request flow

1. Fork the repo and create a feature branch from `main`. Do not push
   to `main` directly.
2. Run `make check` before pushing. CI will repeat all gates plus
   `govulncheck` and a Docker build.
3. Open the PR against `main`. Fill in the PR template; tick the
   checklist boxes only when the work is actually done.
4. Squash-merge is the default; keep the PR title in Conventional Commit
   form so the merged commit subject reads cleanly.
5. New behaviour requires both code change and doc change in the same PR.

## Reporting issues

Use the bug-report or feature-request issue template. For security
issues, do not open a public issue — see `SECURITY.md` for the private
disclosure process.
